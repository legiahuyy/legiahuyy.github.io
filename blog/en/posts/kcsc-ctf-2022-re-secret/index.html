KCSC CTF 2022: secret [RE]

This post is a write-up for the challenge “secret” from KCSC-CTF 2022 which I just participated on Sunday, last week. The challenge seems to be a mixed of miscellaneous and reverse expertise as it requires us to analyze both a network PCAP file and an executable in order to obtain the flag.

You can download the challenge here.

Footprinting⌗

The target’s directory contains two following files, capture.pcapng and secret.exe.

secret.exe⌗

Fortunately, the result provided from ExeinfoPE shows that the binary file is not packed and, presumably, obfuscated by any third-parties. Yet, keep in mind that almost every reversing challenge usually has its own implementation of code obfuscation or data encryption, which we need to solve later.

Based on the result, we can tell there is also no hidden executable in the original one so it is not necessary to scan and dump secret.exe’s resource.

capture.pcapng⌗

The pcapng file has no unusual characteristics or properties, it shows us the conversation between 192.168.123.111 and 192.168.123.123 and based on the context, the host with the fourth octet 111 is sending assumingly encrypted data to the other one.

Reversing⌗

Examine the binary with IDA, pseudo code of its main function as follows:

As demonstrated, this is apparently an initialization for a WinSock connection using those familiar API calls like WSAStartup, htons, inet_addr, … straight from the book.

The function does:

  • Create a singleton mutex object with default namespace as 9_5_diem, not that important

  • Initialize sockaddr_in struct as:

    sockaddr_in.sin_family = 2 // AF_INET
sockaddr_in.sin_addr.s_addr = inet_addr("192.168.123.123")
sockaddr_in.sin_port = htons(1337) // 0x593

    and then call socket(AF_INET, SOCK_STREAM, IPPROTO_TCP) with address family specification as IPv4, socket type as byte stream and protocol as TCP, respectively

  • Establish the connection using connect with the aforementioned struct as parameter

  • Call sub_401983 and release the mutex then finally exit.

Nothing much of an interest until we jump into sub_401983 which actually does something less mundane:

In detail, sub_401983 does:

  • Create two sections of shared memory (called pipes) that it would later use for communication, and SetHandleInformation is to ensure the memory regions is not inherited.

  • Initialize ProcessInformation and StartupInfo to redirect STDIN and STDOUT to the above regions

  • Call cmd.exe via CreateProcessW then execute StartAddress and sub_4018B3 simultaneously

  • cmd.exe should receive inputs from the pipes, execute them and redirect its STDOUT/STDERR to the pipes

  • Wait indefinitely (WaitForSingleObject) for cmd.exe to exit and return

In simple term, the function basically creates anonymous pipes for both STDIN and STDOUT, it then calls cmd.exe and creates a thread pool consists of StartAddress and sub_4018B3, waits for them and exits eventually.

Next, we would want to have a look at StartAddress’s pseudo C:

  }
  return result;
